# 设计文档：Codex 微信文件回传

**日期：** 2026-03-26  
**状态：** Draft Approved  
**范围：** 支持 `Codex` 在微信会话中，在用户明确提出“发送给我”这类诉求时，将本地生成的文件回传给当前微信用户。  
**主实现路径：** 复用 `openclaw-weixin` 现有的出站文件/媒体发送能力，而不是在 v1 里直接扩展当前仓库内置的文本型微信桥接实现。

---

## 一、概述

AionUi 当前已经支持微信中的 Codex 会话，但现有仓库内置微信通道本质上仍是“文本桥”，不能把 Codex 生成的本地文件直接回传给用户。与此同时，运行时接入的 `openclaw-weixin` 已经具备可工作的本地文件上传与发送能力。

本设计文档定义一条分阶段、带测试闸门的实现路线，使 Codex 能在微信会话中，在用户以自然语言明确表达“把文件发给我”时，把本地生成的文件回传给当前微信用户。每个阶段都必须先通过对应测试，再进入下一阶段。

这份文档的目标是成为后续 agent 的单一事实来源，明确说明：

1. 这个需求为什么存在。
2. 选择了哪条实现路径，以及为什么不选其他路径。
3. 每个阶段要做什么、怎么验收、什么时候算通过。
4. 当前已经做到哪一步，后续接手时如何避免跑偏。

---

## 二、背景

### 2.1 用户问题

在微信会话中，Codex 可以在本地工作区生成文件，但目前无法把这些文件通过微信直接回传给用户。这会破坏以下典型工作流：

- “生成一份报告，然后发给我”
- “把 PDF 做出来，再传回这个微信聊天”
- “导出结果文件，并发到当前会话里”

用户希望最终达到的效果是：

1. Codex 能理解“生成文件”和“把文件发给我”之间的差异。
2. Codex 能把生成好的文件发回当前微信用户，而不是只输出路径或文字说明。
3. 实现过程有清晰的分阶段设计、测试门槛和状态跟踪，后续 agent 接手时不会丢上下文。

### 2.2 当前技术现状

仓库内置的微信通道目前仍是文本型桥接：

- [WeixinAdapter.ts](/E:/GithubProj/AionUi/src/process/channels/plugins/weixin/WeixinAdapter.ts#L14)
- [WeixinPlugin.ts](/E:/GithubProj/AionUi/src/process/channels/plugins/weixin/WeixinPlugin.ts)

当前 `weixin -> ChannelMessageService -> Codex` 这条链路只向 Codex 传递纯文本内容：

- [ChannelMessageService.ts](/E:/GithubProj/AionUi/src/process/channels/agent/ChannelMessageService.ts#L213)

但运行时的 `openclaw-weixin` 插件已经具备完整的本地文件上传与发送流程：

- [send-media.ts](C:/Users/jam/AppData/Roaming/AionUi/aionui/codex-temp-1774456119934/pkg_unpack_plugin_201/package/src/messaging/send-media.ts#L17)
- [send-media.ts](C:/Users/jam/AppData/Roaming/AionUi/aionui/codex-temp-1774456119934/pkg_unpack_plugin_201/package/src/messaging/send-media.ts#L56)

因此，当前的真实问题不是“微信底层完全不会发文件”，而是“现有 AionUi 会话链路还没有把这项能力以安全、可控、可被 Codex 调用的方式暴露出来”。

## 三、目标

在微信会话中，当且仅当用户以自然语言明确表达“希望把文件发给我”时，允许 Codex 将本地生成的文件回传给当前微信用户。

### 3.1 成功标准

1. 用户可以在微信里要求 Codex 生成文件并发回当前聊天。
2. Codex 可使用的发送能力只能指向当前微信会话用户。
3. 常见文件类型如 `txt`、`md`、`pdf`、`png`、`zip` 至少能覆盖。
4. 每次发送尝试都有日志和可追踪结果。
5. 整个功能按阶段推进，每个阶段都有测试和通过门槛。

### 3.2 非目标

1. v1 不支持任意指定收件人。
2. v1 不支持“只说生成文件”时自动发送。
3. v1 不重写整个内置微信通道。
4. v1 不把 skill 当成真正的传输层。
5. v1 不尝试一次性抽象成所有渠道通用的自然语言发送框架。

---

## 四、已确认的产品决策

以下决策已在设计阶段确认，v1 视为冻结：

1. **底层能力来源：** 复用 `openclaw-weixin` 的现有文件发送能力。
2. **触发条件：** 只有用户明确要求发送时才允许发送。
3. **意图识别方式：** 第一版按自然语言意图识别，而不是只支持固定命令白名单。

---

## 五、为什么选择这条方案

### 5.1 推荐方案

在 AionUi 后端增加一个专门的“微信文件投递能力层”，内部复用 `openclaw-weixin` 的文件发送逻辑，再把这项能力以一个窄接口的 channel-aware tool/action 暴露给 Codex，且这个工具只能作用于当前微信会话对应的用户。

这条路径最合适，原因如下：

1. 复用了已经可工作的文件发送底层。
2. 避免在第一版直接改造整条内置微信文本桥。
3. 给 Codex 的能力边界清晰，不是无限制的外发能力。
4. 能把“意图识别”“工具暴露”“发送执行”“审计日志”拆成清楚的层次。

### 5.2 不选的方案

#### 方案 A：直接把内置微信插件改造成完整媒体通道

不作为 v1 主路线。原因是当前内置实现还是文本 promise bridge，若直接补齐媒体/文件出站，改动会扩散到更多桥接层，不如先复用已经存在的 `openclaw-weixin` 文件发送通路。

#### 方案 B：只靠 skill 实现文件发送

不成立。skill 可以引导 agent 何时调用发送能力，但不能凭空创造后端运行时没有的发送 API。

#### 方案 C：先做成全渠道通用文件发送工具

不作为 v1。当前用户目标非常明确，就是“当前微信会话用户文件回传”。先做成微信专用、当前会话绑定的能力更稳。

## 六、功能需求

### FR-1 当前会话用户回传

Codex 只能把文件发送给当前微信会话用户。收件目标必须来自当前会话上下文，不能由模型任意生成。

### FR-2 支持本地绝对路径

发送链路必须接受本地绝对路径，且该路径通常由 Codex 或相关工具在当前任务中生成。

### FR-3 必须有明确发送意图

只有当用户当前轮消息中明确表达“发给我”“传回来”“发到微信”之类的诉求时，Codex 才允许触发发送工具。

允许触发的示例：

- “生成后发给我”
- “把文档发到微信给我”
- “做好以后把文件传回来”
- “导出 pdf 然后发我”

不允许自动发送的示例：

- “帮我生成一份报告”
- “导出一个 pdf”
- “保存到本地就行”

### FR-4 失败必须显式可见

如果发送失败，不能静默吞掉。Codex 需要拿到明确的失败结果，并向用户解释失败原因。

### FR-5 发送行为可审计

每次发送尝试至少要记录以下信息：

- conversation id
- channel/platform
- 当前微信目标用户 id
- 文件路径
- 文件类型
- 时间戳
- 成功/失败
- 失败原因

### FR-6 必须按阶段推进

整个功能实现必须分阶段进行。每个阶段都要有必测项和通过条件，未通过前不得进入下一阶段。

## 七、非功能需求

### NFR-1 安全性

v1 不允许 Codex 任意选择收件人。

### NFR-2 确定性

发送工具的协议必须窄而稳定。模型只提供文件路径和可选说明；收件用户和账号解析必须来源于运行时上下文。

### NFR-3 可调试性

发送失败时，必须能通过日志快速定位原因，而不是依赖人工调试器。

### NFR-4 可接续性

后续 agent 只要读这份文档，就应当立刻知道：

1. 这个需求的背景是什么。
2. 当前确认的架构是什么。
3. 还没做的内容有哪些。
4. 每个阶段的测试标准是什么。
5. 现在已经做到哪一步。

## 九、建议架构

## 9.1 高层结构

这个功能建议拆成四层：

1. **意图闸门层**
   判断用户当前消息是否明确要求“把文件发回来”。

2. **Codex 工具暴露层**
   只在微信会话中暴露一个窄接口工具给 Codex。

3. **微信文件投递服务层**
   在 AionUi 后端实现 `WeixinFileDeliveryService`，负责校验输入、解析当前微信上下文、调用复用后的 `openclaw-weixin` 文件发送路径。

4. **审计与状态层**
   记录每次发送结果，并把结构化成功/失败信息回给 Codex。

### 9.2 目标数据流

```text
[微信用户消息]
  -> 内置微信通道
  -> ChannelMessageService
  -> Codex 会话
  -> Codex 判断：
       “用户当前明确要求把文件发给他”
  -> Codex 调用 channel-aware 文件发送工具
  -> WeixinFileDeliveryService
       -> 校验本地绝对路径
       -> 解析当前微信会话对应用户
       -> 解析正确的发送账号/运行时绑定
       -> 调用 openclaw-weixin 出站文件发送能力
  -> 返回结构化结果给 Codex
  -> Codex 在聊天中确认成功或报告失败
```

## 十、组件设计

### 10.1 `WeixinFileDeliveryService`

新增后端服务，负责真正执行微信文件发送。

#### 职责

1. 校验输入路径。
2. 确保路径是绝对路径。
3. 确保目标存在且是普通文件。
4. 从当前会话上下文解析微信目标用户。
5. 解析正确的微信发送账号和运行时绑定。
6. 调用复用后的 `openclaw-weixin` 文件发送逻辑。
7. 返回结构化成功/失败结果。
8. 产出审计日志。

#### 非职责

1. 不负责自然语言理解。
2. 不允许模型自由选择收件人。
3. 不在这里做开放式策略判断，只做执行层校验。

#### 建议接口

```typescript
interface SendFileToCurrentWeixinUserParams {
  conversationId: string;
  filePath: string;
  caption?: string;
}

interface SendFileToCurrentWeixinUserResult {
  success: boolean;
  channel: 'weixin';
  deliveredTo?: string;
  accountId?: string;
  messageId?: string;
  errorCode?: string;
  errorMessage?: string;
}
```

### 10.2 Codex 工具暴露层

只在微信会话中向 Codex 暴露一个窄工具。

#### 建议工具协议

```text
send_file_to_current_weixin_user(
  file_path: string,
  caption?: string
)
```

#### 工具约束

1. 只能在微信会话中可见或可调用。
2. 只能发给当前会话用户。
3. 必须要求绝对路径。
4. 必须返回结构化错误。

### 10.3 意图闸门层

自然语言意图控制应发生在工具规划前或规划过程中。

#### 必须满足的规则

只有当用户当前轮明确表达“把文件发给我”“发回这个微信聊天”等诉求时，才允许进入发送工具。

#### 设计原则

第一版应保守实现，宁可漏判，也不要误发。

### 10.4 审计日志

每次发送尝试都应该落一条结构化日志。

建议字段：

```typescript
{
  feature: 'codex_weixin_file_delivery',
  conversationId,
  source: 'weixin',
  targetUserId,
  accountId,
  filePath,
  mimeType,
  success,
  errorCode,
  errorMessage,
  timestamp,
}
```

## 十一、分阶段实施方案

必须严格按阶段推进。当前阶段测试通过前，不得进入下一阶段。

### Phase 0：设计冻结与基线建立

**目的**

在正式开发前冻结需求范围、架构和测试门槛。

**要求产出**

1. 本地设计文档已存在。
2. 语雀设计文档已同步。
3. 当前实现状态已记录。
4. 后续实现必须引用本设计文档。

**测试 / 校验**

1. 本地 spec 文件存在于约定路径。
2. 语雀文档已存在于目标知识库。
3. 文档中必须包含：
   - 背景
   - 目标
   - 非目标
   - 方案选择
   - 分阶段设计
   - 测试门槛
   - 当前状态表

**通过标准**

本地与远端文档都存在，并经用户确认后，Phase 0 才算完成。

### Phase 1：后端投递能力打通

**目的**

实现 `WeixinFileDeliveryService`，在当前微信会话上下文下，复用 `openclaw-weixin` 文件发送能力，把本地文件发给当前微信用户。

**要求产出**

1. 新服务模块完成。
2. 当前微信会话上下文解析完成。
3. 复用 `openclaw-weixin` 出站发送能力的稳定接入边界完成。
4. 结构化结果与审计日志完成。

**必测项**

1. 单测：拒绝非绝对路径。
2. 单测：拒绝不存在文件。
3. 单测：拒绝非普通文件路径。
4. 单测：能从会话上下文解析当前微信目标用户。
5. 单测：无法解析微信运行时/账号时返回结构化错误。
6. 集成测试：使用 mock transport 完成一次本地文件发送流程。
7. 手测：在受控环境中，至少有一个真实本地文件能发到当前微信用户。

**通过标准**

自动化测试全部通过，且真实文件手测通过后，Phase 1 才可结束。

### Phase 2：Codex 工具接入

**目的**

把文件发送能力接入 Codex，形成微信会话下可调用的窄工具。

**要求产出**

1. 工具注册或等效能力暴露完成。
2. 该工具只在微信会话中可用。
3. 工具自动绑定当前会话上下文。
4. 工具内部调用 Phase 1 的后端服务。

**必测项**

1. 单测：非微信会话下工具不可用。
2. 单测：工具强制校验绝对路径。
3. 单测：成功调用返回结构化 success 结果。
4. 单测：失败调用返回结构化 error 结果。
5. 端到端手测：
   - 用户要求生成并发送文件
   - Codex 生成文件
   - Codex 调工具
   - 微信收到文件
   - Codex 回答发送成功

**通过标准**

真实微信会话端到端验证通过，且自动化测试通过后，Phase 2 才算完成。

### Phase 3：显式发送意图闸门

**目的**

只在用户当前轮明确要求发送时允许工具进入候选路径。

**要求产出**

1. 保守型自然语言意图规则完成。
2. 提示词 / 工具指导更新，确保 Codex 只在明确发送意图下使用该工具。
3. “只生成、不发送”的负路径行为明确。

**必测项**

1. 正例意图能触发工具资格。
2. 负例意图不能触发工具资格。
3. 模糊表达默认不发送。
4. 手测：
   - “生成报告”不会自动发送
   - “生成报告并发给我”会发送

**通过标准**

正反例行为与设计一致，且手测确认不会明显误发。

### Phase 4：加固、可观测性与恢复能力

**目的**

让功能达到可维护、可追踪、可恢复的生产级状态。

**要求产出**

1. 审计日志字段完整。
2. 失败类型归类稳定。
3. 用户侧失败提示可理解。
4. 重复发送与边界情况行为明确。

**必测项**

1. 失败日志足以定位问题。
2. 工具 / 网络 / 运行时失败都能返回结构化错误。
3. 重复调用处理已测试。
4. 手动失败注入：
   - 文件缺失
   - 发送账号不可用
   - 传输失败

**通过标准**

失败可被观察、可被解释、不会静默吞掉后，Phase 4 才算完成。

## 十三、风险

### 风险 1：上下文解析漂移

AionUi 内置微信会话上下文与复用的 `openclaw-weixin` 运行时上下文未必天然一致，可能导致发送失败或账号解析错误。

**缓解方式**

明确做上下文映射逻辑，并用真实会话数据测试。

### 风险 2：自然语言意图误判导致误发

自然语言识别容易把“生成文件”误判成“生成并发送”。

**缓解方式**

v1 采用保守策略，宁可不发，也不要误发。

### 风险 3：依赖临时运行时路径

当前分析到的 `openclaw-weixin` 源码位于运行时解包目录，这不一定是稳定集成边界。

**缓解方式**

真正实现时必须找到稳定、可持续的运行时接入抽象，而不是硬编码临时目录。

### 风险 4：审计不足

如果没有结构化日志，后续 agent 很难区分失败到底发生在意图层、工具层、上下文层还是传输层。

**缓解方式**

在 Phase 1 就建立审计日志，并在 Phase 4 做完整性验证。
